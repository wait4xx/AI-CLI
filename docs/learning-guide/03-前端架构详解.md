# 第一章：React 基础到进阶

在深入项目代码之前，我们先系统性地梳理 React 的核心概念。这些知识是理解后续所有代码的基础。

## 1.1 JSX 语法深入

### 1.1.1 JSX 的本质

JSX（JavaScript XML）是 React 中用于描述 UI 的语法扩展。它并不是一种模板语言，而是 `React.createElement()` 函数调用的语法糖。当编译器（如 Babel、TypeScript）处理 JSX 时，会将其转换为普通的 JavaScript 函数调用。

例如，以下 JSX 代码：

```tsx
<div className="container">
  <h1>标题</h1>
  <p>内容</p>
</div>
```

会被编译为：

```javascript
React.createElement(
  'div',
  { className: 'container' },
  React.createElement('h1', null, '标题'),
  React.createElement('p', null, '内容')
)
```

在 React 17+ 中，引入了新的 JSX 转换（`react-jsx`），不再需要显式导入 `React`。编译器会自动从 `react/jsx-runtime` 导入必要的函数。这就是为什么在本项目的 `tsconfig.json` 中配置了 `"jsx": "react-jsx"`。

### 1.1.2 JSX 中的表达式

JSX 使用花括号 `{}` 嵌入 JavaScript 表达式。注意，只能嵌入**表达式**（expression），不能嵌入**语句**（statement）。

```tsx
// ✅ 正确 — 表达式
const name = "World"
<h1>Hello, {name}</h1>
<h1>{1 + 2}</h1>
<h1>{condition ? "A" : "B"}</h1>
<h1>{items.map(i => <span>{i}</span>)}</h1>

// ❌ 错误 — 语句
<h1>{if (condition) { "A" }}</h1>  // 语法错误
<h1>{const x = 1}</h1>            // 语法错误
```

### 1.1.3 JSX 属性

JSX 中的属性使用驼峰命名（camelCase），这与原生 HTML 属性不同：

```tsx
// HTML → JSX 的映射
// class → className
// for → htmlFor
// onclick → onClick
// onchange → onChange
// tabindex → tabIndex
// autocomplete → autoComplete

<input
  type="text"
  value={username}
  onChange={(e) => setUsername(e.target.value)}
  className="input-field"
  autoComplete="username"
/>
```

在本项目的 `App.tsx` 中，登录表单就使用了这些属性：

```tsx
<input
  type="text"
  placeholder="用户名"
  value={username}
  onChange={(e) => setUsername(e.target.value)}
  className="w-full px-3 py-2 rounded bg-dark-surface border border-dark-border text-gray-100 text-sm focus:outline-none focus:border-blue-500"
  autoComplete="username"
/>
```

这里 `value` 和 `onChange` 形成了一个**受控组件**（Controlled Component），React 完全控制输入框的值。每当用户输入一个字符，`onChange` 事件触发，调用 `setUsername` 更新状态，React 重新渲染组件，将新的 `value` 写回输入框。这个循环确保 React 始终知道输入框的当前值。

### 1.1.4 条件渲染

在 React 中有多种条件渲染的方式：

**方式一：三元运算符**

```tsx
{isAuthenticated ? <Dashboard /> : <LoginForm />}
```

**方式二：逻辑与运算符**

```tsx
{error && <p className="error">{error}</p>}
```

**方式三：提前返回**

```tsx
function Component() {
  if (!data) return <Loading />
  return <Content data={data} />
}
```

在本项目的 `App.tsx` 中，同时使用了这三种方式：

```tsx
// 提前返回 — 根据认证状态切换界面
if (!isAuthenticated) {
  return (
    <div className="h-screen w-screen flex flex-col bg-dark-bg">
      <LoginForm onLogin={login} />
    </div>
  )
}

// 逻辑与 — 错误信息条件显示
{error && (
  <p className="text-red-400 text-xs text-center">{error}</p>
)}

// 条件渲染 — 编辑器懒加载
{editorFile && (
  <Suspense fallback={...}>
    <CodeEditor ... />
  </Suspense>
)}
```

### 1.1.5 列表渲染

使用 `Array.map()` 方法渲染列表，每个列表项必须提供唯一的 `key` 属性：

```tsx
{sessions.map((session, index) => (
  <button key={session.id} ...>
    {session.id.slice(0, 8)}
  </button>
))}
```

`key` 的作用是帮助 React 的协调算法（Reconciliation）识别哪些元素发生了变化、被添加或被删除。它应该是稳定、唯一的标识符。使用数组索引作为 `key` 在列表项可能发生重排时会导致渲染问题，所以最好使用数据本身的唯一标识。

### 1.1.6 事件处理

React 的事件处理使用驼峰命名，并传递一个函数：

```tsx
// 内联函数
<button onClick={() => handleClick(index)}>Click</button>

// 提取的处理函数
<button onClick={handleClick}>Click</button>

// 带参数的处理函数（使用箭头函数包装）
<button onClick={() => handleTabPress(index)}>Tab</button>
```

在 React 中，事件处理函数中的 `e` 是一个合成事件（SyntheticEvent），它是对原生事件的跨浏览器包装。如果需要访问原生事件，可以使用 `e.nativeEvent`。

在本项目的 `SessionTabs.tsx` 中，实现了长按检测：

```tsx
onMouseDown={() => handleTabLongPressStart(index)}
onMouseUp={handleTabLongPressEnd}
onMouseLeave={handleTabLongPressEnd}
onTouchStart={() => handleTabLongPressStart(index)}
onTouchEnd={handleTabLongPressEnd}
```

这里同时处理了鼠标事件和触摸事件，确保在桌面端和移动端都能正常工作。

### 1.1.7 Fragment 语法

当需要返回多个元素但不想添加额外的 DOM 节点时，使用 Fragment：

```tsx
// 完整语法
<React.Fragment>
  <ChildA />
  <ChildB />
</React.Fragment>

// 简写语法
<>
  <ChildA />
  <ChildB />
</>
```

在 `StatusBar` 组件的 `actionsSlot` 属性中，使用了 Fragment 包裹多个操作按钮：

```tsx
<StatusBar actionsSlot={
  <>
    <FileExplorer onFileSelect={...} />
    <SettingsDrawer trigger={...} />
  </>
} />
```

### 1.1.8 TypeScript 与 JSX 的结合

在 TypeScript 中使用 JSX 时，可以为组件的 props 定义类型：

```tsx
interface Props {
  name: string
  age?: number        // 可选属性
  children?: React.ReactNode  // 子元素
}

function Greeting({ name, age, children }: Props) {
  return (
    <div>
      <h1>Hello, {name}</h1>
      {age && <p>Age: {age}</p>}
      {children}
    </div>
  )
}
```

本项目中大量使用了这种模式，例如：

```tsx
interface ConnectionOverlayProps {
  phase: 'DISCONNECTED' | 'CONNECTING_TERM' | 'CONNECTING_CTRL' | 'CONNECTED'
  reconnectCount: number
  cachedScreen?: string
}

export function ConnectionOverlay({ phase, reconnectCount, cachedScreen }: ConnectionOverlayProps) {
  // ...
}
```

这里使用了**字面量联合类型**来限制 `phase` 的取值，提供了比简单 `string` 更强的类型约束。TypeScript 编译器会在编译时检查所有可能的值，确保不会遗漏分支。

## 1.2 组件设计模式

### 1.2.1 函数组件 vs 类组件

在现代 React 开发中，函数组件（Function Component）已经成为主流。本项目全部使用函数组件。

**函数组件的特点：**

```tsx
// 函数组件 — 简洁、易于测试
function StatusBar({ actionsSlot }: { actionsSlot?: React.ReactNode }) {
  const { connectionPhase, agentStatus, sessionId } = useSessionStore()
  // ...
  return <div>...</div>
}
```

**类组件（历史遗留，了解即可）：**

```tsx
// 类组件 — 较冗长，但支持生命周期方法
class StatusBar extends React.Component<Props, State> {
  componentDidMount() { /* ... */ }
  render() { return <div>...</div> }
}
```

函数组件配合 Hooks 可以实现类组件的所有功能，且代码更简洁、逻辑复用更灵活。

### 1.2.2 组件组合模式

React 推崇**组合优于继承**（Composition over Inheritance）的设计理念。本项目中大量使用了组合模式：

**插槽模式（Slot Pattern）：**

```tsx
// StatusBar 接收 actionsSlot 作为"插槽"
export function StatusBar({ actionsSlot }: { actionsSlot?: React.ReactNode }) {
  return (
    <div className="...">
      {/* 其他内容 */}
      <span className="flex-1" />
      {actionsSlot}
    </div>
  )
}

// 使用时传入插槽内容
<StatusBar actionsSlot={
  <>
    <FileExplorer onFileSelect={...} />
    <SettingsDrawer trigger={...} />
  </>
} />
```

这种模式类似于 Vue 的 `slot` 或 Web Components 的 `<slot>` 元素，允许父组件向子组件注入任意内容。

**Render Props 模式：**

```tsx
// 子组件通过回调函数将数据传回父组件
interface FileExplorerProps {
  onFileSelect: (path: string, content: string, language: string) => void
}
```

在 `App.tsx` 中，`FileExplorer` 通过 `onFileSelect` 回调将选中的文件信息传回父组件。

### 1.2.3 组件分层架构

本项目的组件按照职责分为以下几层：

```
┌─────────────────────────────────────┐
│           容器层 (Container)          │
│  App.tsx — 路由、认证、全局状态        │
├─────────────────────────────────────┤
│           布局层 (Layout)             │
│  StatusBar, SessionTabs              │
├─────────────────────────────────────┤
│           功能层 (Feature)            │
│  TerminalContainer, FileExplorer,    │
│  CodeEditor, SettingsDrawer          │
├─────────────────────────────────────┤
│           基础层 (Primitive)          │
│  ConnectionOverlay, QuickActionsPanel│
├─────────────────────────────────────┤
│           工具层 (Utility)            │
│  Hooks, Store, Adapters, Libs        │
└─────────────────────────────────────┘
```

**容器层**负责应用级别的逻辑，如路由切换、认证状态管理。

**布局层**负责页面结构的组织，如顶部状态栏、标签页栏。

**功能层**包含核心业务组件，每个组件对应一个完整的功能模块。

**基础层**是较小的、可复用的 UI 组件。

**工具层**提供非 UI 的逻辑复用，如自定义 Hook、状态管理、适配器等。

### 1.2.4 受控与非受控的混合使用

在实际项目中，受控和非受控模式经常混合使用。例如 `TerminalContainer` 中：

- xterm.js Terminal 实例是一个**非受控**的第三方库，它自己管理终端的渲染状态
- 但 React 通过 `useRef` 持有对它的引用，并通过 `useEffect` 同步外部状态（如 `fontSize`、`theme`）到 Terminal 实例

```tsx
// 非受控 — Terminal 实例通过 ref 持有
const termRef = useRef<Terminal | null>(null)

// 受控 — 通过 useEffect 将 store 中的 fontSize 同步到 Terminal
useEffect(() => {
  if (termRef.current) {
    termRef.current.options.fontSize = fontSize
    fitAddonRef.current?.fit()
  }
}, [fontSize])
```

这种模式在与第三方库集成时非常常见。

## 1.3 React Hooks 全面解析

### 1.3.1 useState — 状态管理

`useState` 是最基本的 Hook，用于在函数组件中添加状态：

```tsx
const [state, setState] = useState(initialValue)
```

**基本用法：**

```tsx
// App.tsx 中的登录表单状态
const [username, setUsername] = useState('')
const [password, setPassword] = useState('')
const [error, setError] = useState('')
const [loading, setLoading] = useState(false)
```

每个 `useState` 调用创建一个独立的状态变量和更新函数。React 内部通过调用顺序（第一次 `useState` 调用对应第一个状态，第二次对应第二个状态，以此类推）来维护这些状态。

**函数式更新：**

当新状态依赖旧状态时，使用函数式更新：

```tsx
// ❌ 可能有闭包问题
setReconnectCount(reconnectCount + 1)

// ✅ 函数式更新，确保使用最新值
setReconnectCount((c) => c + 1)
```

在 `useDualChannelWS.ts` 中，重连计数就使用了函数式更新，因为 `setReconnectCount` 可能在闭包中被调用，闭包捕获的 `reconnectCount` 可能不是最新值。

### 1.3.2 useRef — 引用持有

`useRef` 用于持有对 DOM 元素或任意可变值的引用，且更新不会触发重新渲染：

```tsx
const ref = useRef<T>(initialValue)
```

**持有 DOM 元素：**

```tsx
// TerminalContainer.tsx
const containerRef = useRef<HTMLDivElement>(null)

// JSX 中通过 ref 属性绑定
<div ref={containerRef} className="w-full h-full" />
```

**持有可变值（替代实例变量）：**

```tsx
// useDualChannelWS.ts 中大量使用 useRef 持有可变值
const termWsRef = useRef<WebSocket | null>(null)
const ctrlWsRef = useRef<WebSocket | null>(null)
const termPingRef = useRef<ReturnType<typeof setInterval> | null>(null)
const reconnectTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null)
const reconnectDelayRef = useRef(INITIAL_RECONNECT_DELAY)
const isConnectingRef = useRef(false)
```

为什么用 `useRef` 而不是 `useState`？因为这些值的变化不需要触发 UI 重新渲染。WebSocket 实例、定时器 ID、延迟值等都是"幕后"状态，UI 不需要对它们的变化做出反应。使用 `useState` 会导致每次更新都触发重新渲染，造成不必要的性能开销。

**useRef 与闭包问题：**

`useRef` 的一个关键特性是它始终指向最新的值（通过 `.current` 属性），这可以避免闭包中的陈旧引用问题：

```tsx
// useAuth.ts 中使用 ref 避免闭包陈旧
const doRefreshTokenRef = useRef<() => Promise<string>>()
doRefreshTokenRef.current = async () => {
  // 这里始终能访问最新的状态
  const currentRefresh = useSessionStore.getState().refreshToken
  // ...
}

// 定时器回调通过 ref 调用，而不是直接引用函数
refreshTimerRef.current = setTimeout(async () => {
  try {
    await doRefreshTokenRef.current?.()
  } catch {
    // ...
  }
}, refreshIn)
```

如果直接在定时器中引用 `doRefreshToken` 函数，由于闭包捕获，可能会使用过期的函数引用。通过 `useRef`，定时器总是调用最新的函数版本。

### 1.3.3 useCallback — 函数记忆化

`useCallback` 用于缓存函数引用，避免每次渲染都创建新的函数实例：

```tsx
const memoizedFn = useCallback(() => {
  // 函数体
}, [dependency1, dependency2])
```

在 `TerminalContainer.tsx` 中：

```tsx
const {
  connect, disconnect, sendInput, sendResize, sendQuickAction, sendInjectCode, reconnectCount,
} = useDualChannelWS(
  useCallback(() => useSessionStore.getState().accessToken, []),
  refreshTokenFn,
  logout,
)
```

这里 `getAccessToken` 被 `useCallback` 包裹，依赖数组为空 `[]`，意味着这个函数在组件的整个生命周期中保持同一个引用。

### 1.3.4 useMemo — 值记忆化

`useMemo` 用于缓存计算结果，避免每次渲染都重新计算：

```tsx
const memoizedValue = useMemo(() => expensiveComputation(a, b), [a, b])
```

在 `CodeEditor.tsx` 中：

```tsx
const extensions = useMemo(() => {
  const selectionChangeExt = EditorView.updateListener.of((update: any) => {
    if (update.selectionSet || update.docChanged) {
      const sel = update.state.selection.main
      if (!sel.empty) {
        setSelectedText(update.state.sliceDoc(sel.from, sel.to))
      } else {
        setSelectedText('')
      }
    }
  })
  return [getLanguageExtension(language), EditorView.lineWrapping, selectionChangeExt]
}, [language])
```

这里 `extensions` 数组只在 `language` 变化时重新计算。如果不使用 `useMemo`，每次渲染都会创建新的扩展数组，导致 CodeMirror 编辑器完全重新初始化。

### 1.3.5 useEffect — 副作用管理

`useEffect` 是处理副作用的主要 Hook，它的基本语法：

```tsx
useEffect(() => {
  // 副作用函数
  return () => {
    // 清理函数
  }
}, [dependencies])
```

**依赖数组的行为：**

```tsx
useEffect(() => { ... })           // 每次渲染后执行
useEffect(() => { ... }, [])       // 仅在挂载后执行一次
useEffect(() => { ... }, [a, b])   // a 或 b 变化后执行
```

### 1.3.6 自定义 Hook

自定义 Hook 是以 `use` 开头的函数，它可以封装可复用的状态逻辑。本项目有两个自定义 Hook：

- `useDualChannelWS` — 封装双通道 WebSocket 的所有逻辑
- `useAuth` — 封装认证、登录、token 管理的逻辑

自定义 Hook 的核心价值是**逻辑复用**。通过将复杂的逻辑抽取到 Hook 中，组件可以专注于 UI 渲染，而 Hook 负责状态管理和副作用。

### 1.3.7 Hook 的使用规则

Hook 有两条必须遵守的规则：

1. **只在函数组件或自定义 Hook 中调用 Hook** — 不能在普通函数、条件语句、循环中调用。
2. **只在顶层调用 Hook** — 不能在 `if`、`for`、嵌套函数中调用。

React 通过调用顺序来匹配 Hook 和状态。如果 Hook 的调用顺序在不同渲染之间发生变化，React 就无法正确匹配状态。

## 1.4 状态管理哲学

### 1.4.1 React 状态管理的层次

React 应用中的状态可以分为多个层次：

**局部状态（Local State）：** 只在单个组件中使用的状态，如 `useState` 管理的表单输入。

**提升状态（Lifted State）：** 提升到父组件，通过 props 传递给子组件，如 `editorFile` 状态在 `App` 中管理。

**全局状态（Global State）：** 使用状态管理库（如 Zustand）管理，多个组件都需要读取和修改，如 `sessionId`、`connectionPhase`、`agentStatus`。

### 1.4.2 Zustand 的选择理由

本项目使用 Zustand 而非 Redux 或 Context API，原因包括：

1. **极简 API** — 不需要 Provider、Reducer、Action 等样板代码
2. **性能优秀** — 可以精确订阅状态片段，避免不必要的重新渲染
3. **TypeScript 友好** — 类型推导自然
4. **中间件支持** — 支持持久化、日志等中间件

## 1.5 副作用与生命周期

### 1.5.1 useEffect 与生命周期的对应关系

```tsx
// componentDidMount — 组件挂载
useEffect(() => { /* 执行一次 */ }, [])

// componentDidUpdate — 状态/属性变化
useEffect(() => { /* 依赖变化时执行 */ }, [dependency])

// componentWillUnmount — 组件卸载
useEffect(() => { return () => { /* 清理操作 */ } }, [])

// componentDidMount + componentDidUpdate + componentWillUnmount
useEffect(() => {
  // 订阅/设置
  return () => { /* 取消订阅/清理 */ }
}, [dependency])
```

### 1.5.2 清理函数的重要性

清理函数在组件卸载或依赖变化前执行，用于释放资源：

```tsx
// TerminalContainer.tsx 中的事件监听清理
useEffect(() => {
  function handleResize() { fitAddonRef.current?.fit() }
  window.addEventListener('resize', handleResize)
  return () => window.removeEventListener('resize', handleResize)
}, [])

// 定时器清理
useEffect(() => {
  const disposable = termRef.current?.onData((data) => { sendInput(data) })
  return () => disposable?.dispose()
}, [sendInput])
```

### 1.5.3 useEffect 的常见陷阱

**陷阱一：缺少依赖** — 使用函数式更新避免闭包问题。

**陷阱二：无限循环** — 对象/数组每次渲染都是新引用，使用 `useMemo` 稳定引用。

**陷阱三：异步副作用** — `useEffect` 的回调不能直接是 `async` 函数，需要在内部定义异步函数。



### 3.1.3 模块级缓存

```tsx
const terminalCache = new Map<string, Terminal>()
```

核心原则：**终端实例永远不主动销毁**。xterm.js 的 Terminal 实例创建成本很高（初始化渲染缓冲区、加载字体等），销毁和重建会带来明显的性能问题。

#### Effect: 初始化终端实例（核心，空依赖数组）

```tsx
useEffect(() => {
  if (!containerRef.current) return
  const container = containerRef.current
  let term: Terminal
  let fitAddon: FitAddon

  // 清理其他会话的终端
  const cacheKey = sessionId || '__default'
  for (const [key, cachedTerm] of terminalCache.entries()) {
    if (key !== cacheKey) {
      cachedTerm.dispose()
      terminalCache.delete(key)
      fitAddonCache.delete(key)
    }
  }

  // 复用或创建终端
  const cached = terminalCache.get(cacheKey)
  if (cached) {
    term = cached
    fitAddon = fitAddonCache.get(cacheKey)!
    container.appendChild(term.element!)
    fitAddon.fit()
  } else {
    term = new Terminal({
      theme: getXtermTheme(theme),
      fontSize,
      cursorBlink: true,
      scrollback: 5000,
      convertEol: true,
    })

    fitAddon = new FitAddon()

    // 渲染器降级策略（ADR-010）：WebGL → Canvas → DOM
    try {
      term.loadAddon(new WebglAddon())
      rendererTypeRef.current = 'webgl'
    } catch {
      try {
        term.loadAddon(new CanvasAddon())
        rendererTypeRef.current = 'canvas'
      } catch (e) {
        console.warn('[Terminal] Both WebGL and Canvas addons failed, using DOM renderer', e)
      }
    }

    term.loadAddon(fitAddon)
    term.loadAddon(new WebLinksAddon())
    term.open(container)
    fitAddon.fit()

    terminalCache.set(cacheKey, term)
    fitAddonCache.set(cacheKey, fitAddon)
  }

  termRef.current = term
  fitAddonRef.current = fitAddon

  // MobileKeyboardAdapter 初始化
  const keyboardAdapter = new MobileKeyboardAdapter(
    (data) => sendInput(data),
    (keyboardHeight) => {
      if (containerRef.current) {
        containerRef.current.style.paddingBottom = `${keyboardHeight}px`
        setTimeout(() => fitAddonRef.current?.fit(), 50)
      }
    },
  )
  keyboardAdapter.attach(container)
  keyboardAdapterRef.current = keyboardAdapter

  // GestureHandler 初始化
  const gestureHandler = new GestureHandler(
    container,
    (delta) => {
      const currentSize = useSessionStore.getState().fontSize
      const newSize = Math.max(MIN_FONT_SIZE, Math.min(MAX_FONT_SIZE, currentSize + delta))
      if (newSize !== currentSize) {
        setFontSize(newSize)
        const t = termRef.current
        if (t) t.options.fontSize = newSize
        fitAddonRef.current?.fit()
      }
    },
    (text) => sendInput(text),
  )
  gestureHandler.onPinchStart = () => keyboardAdapter.setSuppressFocus(true)
  gestureHandler.onPinchEnd = () => keyboardAdapter.setSuppressFocus(false)
  gestureHandler.attach()
  gestureHandlerRef.current = gestureHandler

  // 清理函数（ADR-011: removeChild, never dispose）
  return () => {
    keyboardAdapter.destroy()
    gestureHandler.destroy()
    if (term.element && term.element.parentNode) {
      term.element.parentNode.removeChild(term.element)
    }
  }
}, [])
```

这是整个组件中最复杂的 Effect。空依赖数组 `[]` 意味着只在组件挂载时执行一次。

**渲染器降级策略（ADR-010）：** 按优先级尝试 WebGL → Canvas → DOM，使用 `try/catch` 包裹，确保在任何环境下都能工作。

**手势与键盘的协调：** 当双指缩放开始时，抑制键盘适配器的聚焦行为，防止缩放手势意外触发虚拟键盘。

```tsx
    connect(sessionId, cols, rows, termRef.current)
  }
}, [accessToken, sessionId, isConnected, connectionPhase, connect])

  function handleVisibilityChange() {
    const term = termRef.current
    if (!term || !term.element) return
    if (document.hidden) {
      if (term.element.parentNode) term.element.parentNode.removeChild(term.element)
    } else {
      if (containerRef.current && !term.element.parentNode) {
        containerRef.current.appendChild(term.element)
        fitAddonRef.current?.fit()
      }
    }
  }
  document.addEventListener('visibilitychange', handleVisibilityChange)
  return () => document.removeEventListener('visibilitychange', handleVisibilityChange)
}, [])

// 组件卸载清理
useEffect(() => {
  return () => {
    disconnect()
    for (const [key, cachedTerm] of terminalCache.entries()) {
      cachedTerm.dispose()
      terminalCache.delete(key)
      fitAddonCache.delete(key)
    }
  }
}, [disconnect])
```

每个 Effect 都有明确的职责和清理函数。`onData` 和 `onResize` 的 Effect 会在依赖变化时先清理旧的监听器再添加新的，避免重复监听。

# 第四章：Hooks 逐行详解

## 4.1 useDualChannelWS.ts — 双通道 WebSocket

### 4.1.1 文件概览

这是整个项目中最复杂、最核心的自定义 Hook，约 400 行代码。它管理两个 WebSocket 连接：

1. **终端通道（Terminal Channel）** — 传输终端的输入输出数据（二进制）
2. **控制通道（Control Channel）** — 传输控制消息（JSON），如状态更新、resize、审批等

双通道设计的优势：
- 终端数据流（高频、二进制）不会被控制消息阻塞
- 控制消息可以独立于终端数据处理
- 两个通道可以独立重连，提高可靠性

### 4.1.2 常量定义

```tsx
const WS_BASE = import.meta.env.VITE_WS_URL || `${window.location.protocol === 'https:' ? 'wss' : 'ws'}://${window.location.host}`

const MAX_RECONNECT_DELAY = 30_000    // 最大重连延迟 30 秒
const INITIAL_RECONNECT_DELAY = 1_000 // 初始重连延迟 1 秒
const PING_INTERVAL = 30_000          // 心跳间隔 30 秒
const RESIZE_DEBOUNCE = 200           // resize 防抖 200ms
const RESIZE_THROTTLE = 1_000         // resize 节流 1 秒
```

`WS_BASE` 根据页面协议自动选择 `ws://` 或 `wss://`。

### 4.1.3 状态和引用


### 4.1.4 工具函数

```tsx
function clearAllTimers() {
  if (termPingRef.current) { clearInterval(termPingRef.current); termPingRef.current = null }
  if (ctrlPingRef.current) { clearInterval(ctrlPingRef.current); ctrlPingRef.current = null }
  if (reconnectTimerRef.current) { clearTimeout(reconnectTimerRef.current); reconnectTimerRef.current = null }
  if (resizeDebounceRef.current) { clearTimeout(resizeDebounceRef.current); resizeDebounceRef.current = null }
}

function closeSockets() {
  if (termWsRef.current) {
    termWsRef.current.onopen = null    // 先清除事件处理器
    termWsRef.current.onmessage = null // 防止关闭过程中触发事件
    termWsRef.current.onclose = null
    termWsRef.current.onerror = null
    if (termWsRef.current.readyState === WebSocket.OPEN || termWsRef.current.readyState === WebSocket.CONNECTING) {
      termWsRef.current.close()
    }
    termWsRef.current = null
  }
  // ctrlWsRef 同样处理
}
```

**为什么要先清除事件处理器？** 在 `close()` 被调用后，可能会触发 `onclose` 事件。如果不先清除处理器，可能会在清理过程中触发重连逻辑。

### 4.1.5 重连策略

```tsx
function scheduleReconnect() {
  if (reconnectTimerRef.current) return // 防止重复调度

  const delay = reconnectDelayRef.current
  const jittered = delay * (0.5 + Math.random() * 0.5) // 随机抖动
  reconnectDelayRef.current = Math.min(delay * 2, MAX_RECONNECT_DELAY) // 指数退避

  reconnectTimerRef.current = setTimeout(() => {
    reconnectTimerRef.current = null
    const s = sessionRef.current
    const t = termRef.current
    if (s && t) {
      setReconnectCount((c) => c + 1)
      connectInternal(s.sessionId, s.cols, s.rows, t)
    }
  }, jittered)
}
```

**指数退避 + 随机抖动：** 1s → 2s → 4s → 8s → 16s → 30s（上限），每次延迟在 50%~100% 之间随机化，避免"惊群效应"。

### 4.1.6 独立通道重连

`reconnectTermOnly()` — 终端通道断开但控制通道正常时，只重连终端通道。

`reconnectCtrlOnly()` — 控制通道断开但终端通道正常时，只重连控制通道。

这比完全重连更快、更平滑。

### 4.1.9 核心连接函数

```tsx
function connectInternal(sessionId: string, cols: number, rows: number, term: Terminal) {
  if (isConnectingRef.current) return // 防止并发连接
  isConnectingRef.current = true

  store.getState().setConnected('CONNECTING_TERM')
  const token = getAccessToken()
  if (!token) { isConnectingRef.current = false; onAuthFailure(); return }

  sessionRef.current = { sessionId, cols, rows }
  termRef.current = term
  offlineCacheRef.current = new OfflineCache(sessionId)

  // 终端 WebSocket
  const termWs = new WebSocket(`${WS_BASE}/ws/terminal`)
  termWs.binaryType = 'arraybuffer'
  termWsRef.current = termWs

  termWs.onopen = () => {
    termWs.send(JSON.stringify({ type: 'AUTH', accessToken: token, protocolVersion: PROTOCOL_VERSION }))
  }

  termWs.onmessage = (event) => {
    if (typeof event.data === 'string') {
      const msg = JSON.parse(event.data)
      if (msg.type === 'AUTH_OK') {
        termWs.send(JSON.stringify({ type: 'ATTACH_SESSION', sessionId }))

        // 关键：切换到二进制模式
        termWs.onmessage = (ev) => {
          if (ev.data instanceof ArrayBuffer) {
            const buf = new Uint8Array(ev.data)
            if (buf.length === 1 && buf[0] === 0x01) return // 忽略心跳
            term.write(buf) // 写入终端
            offlineCacheRef.current?.cacheScreen(new TextDecoder().decode(buf))
          }
        }

        startTermPing()
        connectControl(token, sessionId, cols, rows) // 连接控制通道
        return
      }
    }
  }
  // ...
}
```

**消息处理器切换：** 认证阶段消息是 JSON 格式，认证成功后切换为二进制格式。通过在运行时替换 `onmessage` 处理器实现。

**`buf[0] === 0x01`：** 单字节 `0x01` 是服务器发送的心跳包，直接忽略。

### 4.1.10 控制通道连接

```tsx
function connectControl(token, sessionId, cols, rows) {
  store.getState().setConnected('CONNECTING_CTRL')
  const ctrlWs = new WebSocket(`${WS_BASE}/ws/control`)
  ctrlWsRef.current = ctrlWs

  ctrlWs.onopen = () => {
    ctrlWs.send(JSON.stringify({ type: 'AUTH', accessToken: token, protocolVersion: PROTOCOL_VERSION }))
  }

  let authenticated = false
  ctrlWs.onmessage = (event) => {
    const msg = JSON.parse(event.data)
    if (!authenticated) {
      if (msg.type === 'AUTH_OK') {
        authenticated = true
        ctrlWs.send(JSON.stringify({ type: 'INIT_SESSION', sessionId, cols, rows, adapter: 'claude' }))
      }
      return
    }
    if (msg.type === 'SESSION_READY') {
      store.getState().setConnected('CONNECTED')
      store.getState().setSession(sessionId)
      isConnectingRef.current = false
      reconnectDelayRef.current = INITIAL_RECONNECT_DELAY
      setReconnectCount(0)
      startCtrlPing()

      // 重连时刷新离线输入
      if (offlineCacheRef.current?.hasQueuedInputs()) {
        offlineCacheRef.current.flushInputs((data) => sendInput(data))
      }

      // 重连时发送 Ctrl+L 触发 pty 重绘
      if (reconnectCount > 0) {
        const termWs = termWsRef.current
        if (termWs && termWs.readyState === WebSocket.OPEN) termWs.send('\x0c')
      }
      return
    }
    handleCtrlMessage(msg)
  }
}
```

连接流程：AUTH → AUTH_OK → INIT_SESSION → SESSION_READY。重连时发送 `Ctrl+L`（`\x0c`）触发终端重绘。

```tsx
  const newId = crypto.randomUUID()
  const { sessions } = get()
  set({
    sessions: [...sessions, { id: newId, status: 'IDLE', label: newId.slice(0, 8) }],
  })
},

removeSession: (index) => {
  const { sessions, activeSessionIndex } = get()
  if (sessions.length <= 1) return // 至少保留一个
  const newSessions = sessions.filter((_, i) => i !== index)
  let newActiveIndex = activeSessionIndex
  if (index < activeSessionIndex) newActiveIndex = activeSessionIndex - 1
  else if (index === activeSessionIndex) newActiveIndex = Math.min(activeSessionIndex, newSessions.length - 1)
  set({
    sessions: newSessions,
    activeSessionIndex: newActiveIndex,
    sessionId: newSessions[newActiveIndex]?.id ?? null,
  })
},

  const { sessions } = get()
  if (index >= 0 && index < sessions.length) {
    set({
      activeSessionIndex: index,
      sessionId: sessions[index].id,
      agentStatus: sessions[index].status,
    })
  }
},

reset: () => set(initialState),
```

## 6.1 MobileKeyboardAdapter.ts — 移动键盘适配

### 6.1.1 文件概览

`MobileKeyboardAdapter` 是一个纯 TypeScript 类，用于在移动端捕获虚拟键盘输入。它使用隐藏的 `<input>` 元素 + Composition Event 来支持 IME 输入（中日韩文字）。

### 6.1.2 构造函数


**预绑定事件处理器：**

### 6.1.4 IME 输入处理

```tsx
private handleCompositionStart(): void {
  this.isComposing = true
}

private handleCompositionEnd(e: CompositionEvent): void {
  this.isComposing = false
  const text = e.data
  if (text) this.onData(text)
  this.hiddenInput.value = ''
}

private handleInput(e: Event): void {
  if (this.isComposing) return // IME 输入中，忽略

  const inputEvent = e as InputEvent
  if (inputEvent.inputType === 'insertText' && inputEvent.data) {
    this.onData(inputEvent.data)
    this.hiddenInput.value = ''
  } else if (inputEvent.inputType === 'deleteContentBackward') {
    this.onData('\x7f') // DEL 字符
  } else if (inputEvent.inputType === 'insertLineBreak' || inputEvent.inputType === 'insertParagraph') {
    this.onData('\r')
    this.hiddenInput.value = ''
  }
}
```

**Composition Event 流程：**

1. `compositionstart` — IME 输入开始，设置 `isComposing = true`
2. `compositionupdate` — IME 输入中（每次候选词变化）
3. `compositionend` — IME 输入完成，发送最终文本

在 `isComposing` 期间，`handleInput` 忽略所有输入事件，避免将中间状态发送到终端。

## 6.2 GestureHandler.ts — 手势处理

### 6.2.1 文件概览

`GestureHandler` 是一个纯 TypeScript 类，处理两种手势：

1. **双指缩放** — 调整终端字体大小
2. **长按粘贴** — 从剪贴板粘贴文本

### 6.2.2 常量

```tsx
const MIN_FONT_SIZE = 10
const MAX_FONT_SIZE = 32
const PINCH_STEP_PX = 20   // 每 20px 指距变化 = 1px 字体
const LONG_PRESS_MS = 500  // 长按阈值
const MOVE_THRESHOLD = 10  // 长按判定：移动超过此距离取消
```

### 6.2.3 触摸事件处理

```tsx
private handleTouchStart(e: TouchEvent): void {
  if (e.touches.length === 2) {
    // 双指缩放开始
    this.clearLongPressTimer()
    this.initialPinchDistance = this.getDistance(e.touches[0], e.touches[1])
    this.lastPinchDistance = this.initialPinchDistance
    this.onPinchStart?.()
  } else if (e.touches.length === 1) {
    // 可能的长按开始
    this.longPressCancelled = false
    this.longPressStartX = e.touches[0].clientX
    this.longPressStartY = e.touches[0].clientY
    this.longPressTimer = setTimeout(() => {
      if (!this.longPressCancelled) this.handleLongPress()
    }, LONG_PRESS_MS)
  }
}

private handleTouchMove(e: TouchEvent): void {
  if (e.touches.length === 2) {
    // 双指缩放中
    this.clearLongPressTimer()
    const currentDistance = this.getDistance(e.touches[0], e.touches[1])
    const distanceDelta = currentDistance - this.lastPinchDistance
    if (Math.abs(distanceDelta) >= PINCH_STEP_PX) {
      const fontDelta = Math.round(distanceDelta / PINCH_STEP_PX)
      this.onFontSizeChange(fontDelta)
      this.lastPinchDistance = currentDistance
    }
    e.preventDefault()
  } else if (e.touches.length === 1 && this.longPressTimer) {
    // 移动超过阈值取消长按
    const dx = e.touches[0].clientX - this.longPressStartX
    const dy = e.touches[0].clientY - this.longPressStartY
    if (Math.sqrt(dx * dx + dy * dy) > MOVE_THRESHOLD) {
      this.longPressCancelled = true
      this.clearLongPressTimer()
    }
  }
}
```

**双指缩放算法：**

1. 记录初始指距
2. 每次移动时计算指距变化
3. 每 20px 变化触发 1px 字体大小调整
4. `e.preventDefault()` 阻止浏览器默认的缩放行为

**长按检测：**

1. 记录触摸起点
2. 设置 500ms 定时器
3. 如果手指移动超过 10px，取消长按
4. 定时器到期时，尝试从剪贴板读取文本并粘贴

## 6.3 notifications.ts — 通知系统

```tsx
let permissionGranted = false

export async function requestNotificationPermission(): Promise<void> {
  if (!('Notification' in window)) return
  if (Notification.permission === 'granted') {
    permissionGranted = true
    return
  }
  if (Notification.permission !== 'denied') {
    const result = await Notification.requestPermission()

}

export function sendNotification(title: string, body: string): void {

  try {
    const notification = new Notification(title, {
      body,
      icon: '/icon-192.png',

}
```

**设计要点：**

- `sendNotification` 只在页面不可见时发送，避免在用户正在查看页面时弹出通知
- `tag: 'ai-cli-notification'` 确保同一时间只显示一个通知（新通知替换旧通知）

- 使用 `try/catch` 包裹，因为某些环境（如 iframe）可能不支持 Notification 构造函数

## 6.4 offlineCache.ts — 离线缓存

```tsx
const STORAGE_KEY = 'ai_cli_offline_cache'
const MAX_QUEUED_INPUTS = 1000

export class OfflineCache {
  private screenSnapshot: string = ''
  private inputQueue: Array<string | Uint8Array> = []
  private sessionId: string | null = null

  constructor(sessionId?: string) {
    this.sessionId = sessionId ?? null
    this.restore() // 从 sessionStorage 恢复
  }

  cacheScreen(data: string): void {
    this.screenSnapshot = data
    this.persist()
  }

  getCachedScreen(): string {
    return this.screenSnapshot
  }

  queueInput(data: string | Uint8Array): void {
    if (this.inputQueue.length >= MAX_QUEUED_INPUTS) this.inputQueue.shift()
    this.inputQueue.push(data)
    this.persist()
  }

  flushInputs(sendFn: (data: string | Uint8Array) => void): void {
    const queued = [...this.inputQueue]
    this.inputQueue = []
    for (const input of queued) sendFn(input)
    this.persist()
  }

  hasQueuedInputs(): boolean {
    return this.inputQueue.length > 0
  }

  private persist(): void {
    try {
      const serializableInputs = this.inputQueue.filter((i) => typeof i === 'string')
      sessionStorage.setItem(STORAGE_KEY, JSON.stringify({
        screenSnapshot: this.screenSnapshot,
        inputQueue: serializableInputs,
        sessionId: this.sessionId,
        timestamp: Date.now(),
      }))
    } catch { /* sessionStorage 可能已满 */ }
  }

  private restore(): void {
    try {
      const raw = sessionStorage.getItem(STORAGE_KEY)
      if (!raw) return
      const data = JSON.parse(raw)
      if (data.sessionId === this.sessionId || !this.sessionId) {
        this.screenSnapshot = data.screenSnapshot ?? ''
        this.inputQueue = data.inputQueue ?? []
      }
    } catch { /* 忽略解析错误 */ }
  }
}
```

**离线缓存设计：**

- **屏幕快照** — 缓存最后的终端输出，断线时显示在遮罩背景中
- **输入队列** — 断线时的用户输入加入队列，重连后批量发送
- **持久化** — 使用 `sessionStorage`（标签页关闭后自动清除）
- **序列化限制** — 只能持久化字符串输入，`Uint8Array` 无法 JSON 序列化
- **队列上限** — 最多 1000 条，超出时丢弃最早的
- **会话隔离** — 只恢复匹配当前 sessionId 的缓存

# 第七章：xterm.js 深度讲解

## 7.1 xterm.js 简介

xterm.js 是一个用 TypeScript 编写的前端终端模拟器。它可以在浏览器中渲染一个完整的终端界面，支持 ANSI 转义序列、Unicode、256 色和 TrueColor。VS Code 的内置终端就是基于 xterm.js 构建的。

### 7.1.1 核心架构

xterm.js 的架构分为三层：

```
┌─────────────────────────────────┐
│         Terminal API             │  ← 用户接口层
├─────────────────────────────────┤
│         Buffer & Parser          │  ← 数据处理层
│  (InputHandler, BufferService,   │
│   ParserService, CharsetService) │
├─────────────────────────────────┤
│         Renderer                 │  ← 渲染层
│  (DOM / Canvas / WebGL)          │
└─────────────────────────────────┘
```

**Terminal API 层：** 提供用户操作终端的接口，如 `write()`、`writeln()`、`clear()`、`reset()` 等。

**Buffer & Parser 层：** 解析 ANSI 转义序列，维护终端的字符缓冲区（包括主缓冲区和备用缓冲区）。

**Renderer 层：** 将缓冲区的内容渲染到屏幕上。支持三种渲染器：DOM（最慢但兼容性最好）、Canvas（中等）、WebGL（最快）。

### 7.1.2 Terminal 实例

```tsx
const term = new Terminal({
  theme: getXtermTheme(theme),  // 配色方案
  fontSize,                      // 字体大小
  cursorBlink: true,            // 光标闪烁
  scrollback: 5000,             // 回滚缓冲区 5000 行
  convertEol: true,             // \n → \r\n
})
```

**Terminal 构造函数的配置项：**

- **theme** — `ITheme` 类型，定义 16 色 ANSI 调色板、背景色、前景色、光标色等
- **fontSize** — 字体大小（像素）
- **fontFamily** — 字体族（本项目未设置，使用默认的 Menlo, Monaco, monospace）
- **cursorBlink** — 光标是否闪烁
- **scrollback** — 回滚缓冲区大小（行数）。超出的旧内容会被丢弃
- **convertEol** — 是否自动将 `\n` 转换为 `\r\n`
- **allowTransparency** — 是否允许透明背景
- **disableStdin** — 是否禁用标准输入（本项目未使用，因为输入通过 WebSocket 发送）

### 7.1.3 终端生命周期

```tsx
// 1. 创建实例
const term = new Terminal(options)

// 2. 加载插件
term.loadAddon(new FitAddon())
term.loadAddon(new WebglAddon())

// 3. 挂载到 DOM
term.open(containerElement)

// 4. 写入数据
term.write('Hello, World!\r\n')

// 5. 监听事件
const disposable = term.onData((data) => { /* 用户输入 */ })
const disposable2 = term.onResize(({ cols, rows }) => { /* 尺寸变化 */ })

// 6. 调整大小
fitAddon.fit()

// 7. 清理
disposable.dispose()
disposable2.dispose()
term.dispose() // 本项目不调用此方法（ADR-011）
```

### 7.1.4 事件系统

xterm.js 使用 `IDisposable` 模式管理事件订阅：

```tsx
// 订阅事件
const disposable = term.onData((data) => {
  console.log('User typed:', data)
})

// 取消订阅
disposable.dispose()
```

**主要事件：**

- **onData** — 用户输入数据（键盘按键、粘贴等）
- **onResize** — 终端尺寸变化（列数和行数）
- **onRender** — 渲染完成
- **onCursorMove** — 光标移动
- **onLineFeed** — 换行
- **onScroll** — 滚动
- **onBell** — 响铃（`\x07`）
- **onTitleChange** — 标题变化（`\x1b]0;title\x07`）

### 7.1.5 缓冲区

xterm.js 有两个缓冲区：

- **主缓冲区（Normal Buffer）** — 正常的终端输出，支持回滚
- **备用缓冲区（Alternate Buffer）** — 全屏应用（如 vim、less）使用，不支持回滚

应用程序通过转义序列 `\x1b[?1049h` 切换到备用缓冲区，`\x1b[?1049l` 切换回主缓冲区。

## 7.2 Addon 机制

### 7.2.1 Addon 接口

xterm.js 的插件系统基于一个简单的接口：

```typescript
interface ITerminalAddon extends IDisposable {
  activate(terminal: Terminal): void
}
```

每个插件必须实现 `activate` 方法（在插件被加载时调用）和 `dispose` 方法（在插件被卸载时调用）。

### 7.2.2 FitAddon

`FitAddon` 用于自动调整终端的列数和行数，使其适应容器的尺寸。

**工作原理：**

1. 测量容器的宽度和高度
2. 计算单个字符的宽度和高度（通过创建临时测量元素）
3. 用容器尺寸除以字符尺寸，得到列数和行数
4. 调用 `term.resize(cols, rows)` 调整终端尺寸

```tsx
const fitAddon = new FitAddon()
term.loadAddon(fitAddon)

// 手动 fit
fitAddon.fit()

// 获取建议的尺寸
const { cols, rows } = fitAddon.proposeDimensions()
```

**使用场景：**

- 窗口大小变化时
- 虚拟键盘弹出/收起时
- 终端容器大小变化时
- 主题切换后（字体可能变化）

### 7.2.3 WebglAddon

`WebglAddon` 使用 WebGL 2.0 渲染终端内容。它将终端的字符绘制为纹理，通过 GPU 加速渲染。

**优势：**
- 渲染速度最快（GPU 加速）
- 支持更复杂的视觉效果（如透明度、模糊）
- 大量输出时帧率更高

**限制：**
- 需要 WebGL 2.0 支持
- 在某些设备上可能不稳定
- 首次加载需要编译 shader

### 7.2.4 CanvasAddon

`CanvasAddon` 使用 Canvas 2D API 渲染终端内容。它是 WebglAddon 的备用方案。

**优势：**
- 兼容性更好（几乎所有浏览器都支持 Canvas 2D）
- 比 DOM 渲染快

**限制：**
- 比 WebGL 慢
- 不支持某些高级视觉效果

### 7.2.5 WebLinksAddon

`WebLinksAddon` 自动检测终端输出中的 URL，并使其可点击。

**工作原理：**

1. 使用正则表达式匹配 URL 模式
2. 在匹配的 URL 上添加悬停高亮
3. 点击时在新标签页中打开 URL

## 7.3 渲染原理

### 7.3.1 DOM 渲染器

DOM 渲染器是最简单的渲染方式，它直接操作 DOM 元素来显示终端内容。

**原理：**
- 每行终端内容对应一个 `<div>` 元素
- 每个字符对应一个 `<span>` 元素
- 通过 CSS 类名设置颜色和样式

**优点：** 兼容性最好，调试方便
**缺点：** 大量字符时 DOM 节点过多，性能差

### 7.3.2 Canvas 渲染器

Canvas 渲染器使用 Canvas 2D API 绘制终端内容。

**原理：**
- 创建一个 `<canvas>` 元素
- 使用 `ctx.fillText()` 绘制每个字符
- 使用 `ctx.clearRect()` 清除旧内容

**优点：** 比 DOM 渲染快
**缺点：** 需要手动管理绘制逻辑

### 7.3.3 WebGL 渲染器

WebGL 渲染器使用 WebGL 2.0 绘制终端内容。

**原理：**
- 创建一个 `<canvas>` 元素，获取 WebGL 上下文
- 将字符集渲染到纹理图集（Texture Atlas）
- 使用 shader 将纹理绘制到屏幕上
- 只更新变化的字符（脏区域检测）

**优点：** 最快，支持大量字符
**缺点：** 需要 WebGL 支持，首次加载需要编译 shader

### 7.3.4 本项目的渲染策略

```tsx
// ADR-010: 渲染器降级策略
try {
  term.loadAddon(new WebglAddon())    // 优先 WebGL
  rendererTypeRef.current = 'webgl'
} catch {
  try {
    term.loadAddon(new CanvasAddon())  // 备用 Canvas
    rendererTypeRef.current = 'canvas'
  } catch (e) {
    console.warn('[Terminal] Both WebGL and Canvas addons failed, using DOM renderer', e)
    // 最后兜底 DOM 渲染
  }
}
```

按优先级尝试：WebGL → Canvas → DOM。确保在任何环境下都能工作。

# 第九章：PWA 技术

## 9.1 PWA 概述

PWA（Progressive Web App）是一种使用 Web 技术构建的应用，可以像原生应用一样安装到设备上，支持离线工作、推送通知等功能。

### 9.1.1 PWA 的核心要素

1. **Web App Manifest** — 定义应用的元数据（名称、图标、主题色等）
2. **Service Worker** — 后台脚本，拦截网络请求、缓存资源、处理推送通知
3. **HTTPS** — PWA 必须在安全上下文中运行（localhost 除外）

## 9.2 Service Worker

### 9.2.1 Service Worker 的生命周期

```
注册 → 安装 → 等待 → 激活 → 运行
```

1. **注册** — 浏览器发现 Service Worker 脚本并注册
2. **安装** — Service Worker 安装，预缓存指定的资源
3. **等待** — 如果旧的 Service Worker 仍在运行，新的会等待
4. **激活** — 新的 Service Worker 激活，开始控制页面
5. **运行** — 拦截网络请求、处理推送通知等

### 9.2.2 vite-plugin-pwa 的工作方式

`vite-plugin-pwa` 使用 Workbox 库自动生成 Service Worker。在构建时，它会：

1. 扫描构建输出目录中的所有文件
2. 根据 `globPatterns` 配置选择需要预缓存的文件
3. 生成 Service Worker 脚本，包含预缓存清单
4. 生成 Web App Manifest

### 9.2.3 缓存策略

本项目使用 `registerType: 'autoUpdate'`，这意味着：

- Service Worker 在后台自动更新
- 新版本激活后立即接管页面
- 用户无需手动确认更新

**预缓存（Precaching）：**

```tsx
workbox: {
  globPatterns: ['**/*.{js,css,html,ico,png,svg}'],
}
```

所有 JS、CSS、HTML、图标和图片文件在安装阶段就被缓存。这确保了即使在离线状态下，应用也能正常加载。

**运行时缓存（Runtime Caching）：**

本项目未配置运行时缓存策略。API 请求和 WebSocket 连接在离线时会失败，但 UI 仍然可以加载。

```json
{
  "name": "AI-CLI-Mobile",
  "short_name": "AI-CLI",
  "description": "Mobile AI Programming CLI Gateway",
  "theme_color": "#1a1a2e",
  "background_color": "#0f0f1a",
  "display": "standalone",
  "orientation": "any",
  "icons": [
    { "src": "/icon-192.png", "sizes": "192x192", "type": "image/png" },
```

- **name** — 应用的完整名称，显示在安装对话框和启动画面中
- **short_name** — 应用的简称，显示在主屏幕图标下方
- **display: 'standalone'** — 以独立应用模式显示，隐藏浏览器 UI

## 9.4 离线策略

本项目的离线策略分为两层：

**第一层：PWA 预缓存**

- 所有静态资源（JS、CSS、HTML、图标）被预缓存

# 第十章：移动端适配

## 10.1 Viewport 配置

### 10.1.1 Viewport Meta 标签

在 `index.html` 中，viewport meta 标签控制页面在移动端的显示方式：

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no, viewport-fit=cover">
```

- **width=device-width** — 视口宽度等于设备宽度
- **initial-scale=1.0** — 初始缩放比例为 1
- **maximum-scale=1.0** — 禁止用户放大
- **user-scalable=no** — 禁止用户缩放
- **viewport-fit=cover** — 视口延伸到安全区域（刘海屏、圆角等）

### 10.1.2 安全区域

在有刘海屏或底部横条的设备上，`viewport-fit=cover` 使页面延伸到安全区域外。使用 `env()` 函数获取安全区域的内边距：

```css
body {
  padding-top: env(safe-area-inset-top);
  padding-bottom: env(safe-area-inset-bottom);
  padding-left: env(safe-area-inset-left);
  padding-right: env(safe-area-inset-right);
}
```

## 10.2 触摸事件

### 10.2.1 触摸事件类型

- **touchstart** — 手指触摸屏幕
- **touchmove** — 手指在屏幕上移动
- **touchend** — 手指离开屏幕
- **touchcancel** — 触摸被系统取消（如来电）

### 10.2.2 本项目中的触摸事件处理

**SessionTabs.tsx — 长按检测：**

```tsx
onTouchStart={() => handleTabLongPressStart(index)}
onTouchEnd={handleTabLongPressEnd}
```

**GestureHandler.ts — 双指缩放：**

```tsx
this.element.addEventListener('touchstart', this.boundHandlers.touchstart, { passive: false })
this.element.addEventListener('touchmove', this.boundHandlers.touchmove, { passive: false })
this.element.addEventListener('touchend', this.boundHandlers.touchend, { passive: false })
```

注意 `{ passive: false }` — 这允许在事件处理器中调用 `e.preventDefault()`，阻止浏览器的默认触摸行为（如缩放、滚动）。

### 10.2.3 点击延迟

在移动端，浏览器会在 `touchend` 后等待约 300ms 才触发 `click` 事件，以判断用户是否在进行双击缩放。这会导致点击响应延迟。

**解决方案：**

1. 使用 `touchend` 事件替代 `click`
2. 使用 CSS `touch-action: manipulation` 禁用双击缩放
3. 使用 `fastclick` 库（已过时）

本项目同时使用了 `click` 和 `touchend` 事件，在桌面端使用 `click`，在移动端使用 `touchend`。

## 10.3 虚拟键盘

### 10.3.1 虚拟键盘的挑战

在移动端，虚拟键盘的弹出和收起会导致：

1. **视口大小变化** — 键盘占据屏幕空间，可视区域变小
2. **布局重排** — 页面布局需要适应新的视口大小
3. **输入捕获** — 终端需要捕获键盘输入，但 xterm.js 不能直接获取移动端键盘事件

### 10.3.2 MobileKeyboardAdapter 的解决方案

**隐藏输入框：**

```tsx
this.hiddenInput = document.createElement('input')
Object.assign(this.hiddenInput.style, {
  position: 'fixed',
  opacity: '0',
  left: '-9999px',
  width: '1px',
  height: '1px',
  fontSize: '16px', // 防止 iOS Safari 自动缩放
})
```

使用一个隐藏的 `<input>` 元素来接收虚拟键盘的输入。当用户点击终端区域时，自动聚焦这个隐藏输入框，触发虚拟键盘。

**visualViewport 监听：**

```tsx
private handleViewportResize(): void {
  if (!window.visualViewport) return
  const keyboardHeight = window.innerHeight - window.visualViewport.height
  this.onResize(keyboardHeight)
}
```

使用 `window.visualViewport` API 监听视口大小变化。当虚拟键盘弹出时，`visualViewport.height` 会减小，差值就是键盘的高度。

**容器 padding 调整：**

```tsx
(keyboardHeight) => {
  if (containerRef.current) {
    containerRef.current.style.paddingBottom = `${keyboardHeight}px`
    setTimeout(() => fitAddonRef.current?.fit(), 50)
  }
}
```

当键盘弹出时，给终端容器添加底部 padding，防止键盘遮挡终端内容。然后重新 fit 终端，使其适应新的可用空间。

### 10.3.3 iOS Safari 的特殊处理

**自动缩放问题：**

iOS Safari 在输入框字体小于 16px 时会自动缩放页面。解决方案是设置 `fontSize: '16px'`。

**焦点问题：**

iOS Safari 中，`focus()` 调用必须在用户交互的同步上下文中执行。但某些场景下需要延迟调用：

```tsx

}
```

## 10.4 手势处理

### 10.4.1 双指缩放

双指缩放用于调整终端字体大小：

```tsx
private handleTouchMove(e: TouchEvent): void {
  if (e.touches.length === 2) {
    const currentDistance = this.getDistance(e.touches[0], e.touches[1])
    const distanceDelta = currentDistance - this.lastPinchDistance
    if (Math.abs(distanceDelta) >= PINCH_STEP_PX) {
      const fontDelta = Math.round(distanceDelta / PINCH_STEP_PX)
      this.onFontSizeChange(fontDelta)
      this.lastPinchDistance = currentDistance
    }
    e.preventDefault() // 阻止浏览器默认缩放
  }
}
```

**算法：**

1. 计算两个触摸点之间的距离
2. 与上次记录的距离比较，得到变化量
3. 每 20px 变化触发 1px 字体大小调整
4. 阻止浏览器的默认缩放行为

# 第十一章：深入 React 原理

## 11.1 React 的渲染机制

### 11.1.1 Virtual DOM

React 使用 Virtual DOM（虚拟 DOM）来优化 UI 更新。Virtual DOM 是一个轻量级的 JavaScript 对象树，它是真实 DOM 的映射。

**为什么需要 Virtual DOM？**

直接操作 DOM 是昂贵的。每次 DOM 修改都可能触发浏览器的重排（Reflow）和重绘（Repaint）。Virtual DOM 允许 React 在内存中计算最小的 DOM 更新，然后一次性应用到真实 DOM 上。

**Virtual DOM 的工作流程：**

1. **状态变化** — 组件的状态发生变化（如 `setState` 被调用）
2. **重新渲染** — React 调用组件函数，生成新的 Virtual DOM 树
3. **Diff 算法** — React 比较新旧两棵 Virtual DOM 树，找出差异
4. **批量更新** — React 将所有差异一次性应用到真实 DOM 上

```tsx
// 伪代码示意
function reconciler(oldVNode, newVNode) {
  if (oldVNode.type !== newVNode.type) {
    // 类型不同，整个替换
    replaceNode(oldVNode, newVNode)
  } else if (typeof newVNode.type === 'string') {
    // 原生元素，更新属性
    updateProps(oldVNode.props, newVNode.props)
    // 递归处理子节点
    reconcileChildren(oldVNode.children, newVNode.children)
  } else if (typeof newVNode.type === 'function') {
    // 组件，调用 render
    const newChildren = newVNode.type(newVNode.props)
    reconciler(oldVNode.children, newChildren)
  }
}
```

### 11.1.2 Fiber 架构

React 16 引入了 Fiber 架构，它是对 Virtual DOM 的重大改进。

**Fiber 的核心思想：**

- 将渲染工作分解为小的单元（Fiber 节点）
- 每个 Fiber 节点代表一个工作单元
- 可以暂停、恢复和丢弃工作单元
- 优先处理高优先级的更新

**Fiber 节点的数据结构：**

```typescript
interface Fiber {
  tag: number           // 节点类型（函数组件、类组件、原生元素等）
  type: any             // 组件类型或元素类型
  key: string | null    // 唯一标识
  stateNode: any        // DOM 节点或组件实例

  // Fiber 树结构
  return: Fiber | null  // 父 Fiber
  child: Fiber | null   // 第一个子 Fiber
  sibling: Fiber | null // 下一个兄弟 Fiber
  index: number         // 在父 Fiber 中的索引

  // 工作状态
  pendingProps: any     // 新的 props
  memoizedProps: any    // 上次渲染的 props
  memoizedState: any    // 上次渲染的 state

  // 副作用
  flags: number         // 副作用标记
  nextEffect: Fiber | null // 下一个有副作用的 Fiber
}
```

**Fiber 的双缓冲（Double Buffering）：**

React 维护两棵 Fiber 树：

- **current 树** — 当前显示在屏幕上的 Fiber 树
- **workInProgress 树** — 正在构建的新 Fiber 树

当 workInProgress 树构建完成后，React 将其切换为 current 树。这种双缓冲机制确保了渲染过程中的 UI 一致性。

### 11.1.3 调度器（Scheduler）

React 的调度器负责决定何时执行工作。它使用 `requestIdleCallback`（或 polyfill）来在浏览器空闲时执行低优先级的工作。

**优先级队列：**

- **同步优先级（Sync）** — 立即执行，如用户输入
- **连续优先级（Continuous）** — 尽快执行，如动画
- **默认优先级（Default）** — 正常执行，如数据获取
- **低优先级（Low）** — 可以延迟执行
- **空闲优先级（Idle）** — 在浏览器空闲时执行

### 11.1.4 并发特性

React 18 引入了并发特性（Concurrent Features），允许 React 同时准备多个版本的 UI。

**useTransition：**

```tsx
const [isPending, startTransition] = useTransition()

function handleClick() {
  startTransition(() => {
    // 低优先级更新
    setSearchResults(filterData(query))
  })
}
```

`useTransition` 将状态更新标记为低优先级，允许 React 在更新过程中中断以处理高优先级的更新（如用户输入）。

**useDeferredValue：**

```tsx
const deferredQuery = useDeferredValue(query)
```

`useDeferredValue` 创建一个延迟版本的值，允许 React 在空闲时更新这个值。

## 11.2 React 的协调算法

### 11.2.1 Diff 算法

React 的 Diff 算法基于两个假设：

1. **不同类型的元素产生不同的树** — 如果元素类型不同（如 `<div>` 变为 `<span>`），React 会销毁旧树并创建新树。

2. **通过 `key` 标识稳定的元素** — 在列表中，`key` 帮助 React 识别哪些元素是新增、删除或移动的。

**Diff 算法的步骤：**

1. **比较根节点** — 如果类型不同，整个替换
2. **比较属性** — 更新变化的属性
3. **比较子节点** — 使用 `key` 进行匹配
4. **递归处理** — 对匹配的子节点递归执行 Diff

### 11.2.2 Key 的作用

```tsx
// ❌ 不好的 key — 使用数组索引
{items.map((item, index) => (
  <li key={index}>{item.name}</li>
))}

// ✅ 好的 key — 使用唯一标识
{items.map((item) => (
  <li key={item.id}>{item.name}</li>
))}
```

使用数组索引作为 key 在列表项发生重排时会导致问题。例如，如果在列表头部插入一个新项，所有项的索引都会变化，导致 React 认为所有项都发生了变化，触发不必要的重新渲染和 DOM 操作。

### 11.2.3 本项目中的 Key 使用

```tsx
// SessionTabs.tsx
{sessions.map((session, index) => (
  <button key={session.id} ...>
    ...
  </button>
))}

// FileExplorer.tsx
{state.entries.map(entry => (
  <button key={entry.path} ...>
    ...
  </button>
))}

// QuickActionsPanel.tsx
{actions.map((action) => (
  <button key={action.payload} ...>
    ...
  </button>
))}
```

所有列表都使用了稳定的唯一标识作为 key。

## 11.3 React 的事件系统

### 11.3.1 合成事件（SyntheticEvent）

React 实现了自己的事件系统，称为合成事件（SyntheticEvent）。它是对原生浏览器事件的包装，提供了跨浏览器的一致性。

**合成事件的特点：**

1. **跨浏览器一致性** — 不同浏览器的事件行为可能不同，合成事件提供了统一的接口
2. **事件池化（已废弃）** — React 17 之前，合成事件会被复用以减少内存分配
3. **自动绑定** — 在类组件中，事件处理器需要手动绑定 `this`；在函数组件中，不需要
4. **事件委托** — React 将所有事件处理器绑定到根节点上，而不是每个 DOM 元素

### 11.3.2 事件委托

在 React 17 之前，所有事件都委托到 `document` 上。从 React 17 开始，事件委托到 React 树的根 DOM 节点上。

**事件委托的好处：**

1. **减少内存使用** — 只需要一个事件处理器，而不是每个元素一个
2. **自动清理** — 当组件卸载时，事件处理器自动被清理
3. **更好的兼容性** — 可以统一处理不同浏览器的事件差异

### 11.3.3 本项目中的事件处理

```tsx
// 同时处理鼠标和触摸事件
<button
  onClick={() => handleTabPress(index)}
  onMouseDown={() => handleTabLongPressStart(index)}
  onMouseUp={handleTabLongPressEnd}
  onMouseLeave={handleTabLongPressEnd}
  onTouchStart={() => handleTabLongPressStart(index)}
  onTouchEnd={handleTabLongPressEnd}
>
```

本项目同时使用了鼠标事件和触摸事件，确保在桌面端和移动端都能正常工作。

## 11.4 React 的性能优化

### 11.4.1 React.memo

`React.memo` 是一个高阶组件，它对组件的 props 进行浅比较，如果 props 没有变化，跳过重新渲染。

```tsx
const MemoizedComponent = React.memo(function MyComponent({ name, age }) {
  return <div>{name}, {age}</div>
})
```

**本项目未使用 React.memo**，因为所有组件都是函数组件，且状态管理使用 Zustand 的选择器模式来精确控制重新渲染。

### 11.4.2 useMemo 和 useCallback

```tsx
// useMemo — 缓存计算结果
const extensions = useMemo(() => {
  return [getLanguageExtension(language), EditorView.lineWrapping]

// useCallback — 缓存函数引用
const sendInput = useCallback((data) => {
```

### 11.4.3 状态下沉（State Colocation）

将状态放在使用它的最近组件中，而不是提升到父组件。

```tsx

```

本项目中，`username`、`password`、`error`、`loading` 等状态都在 `LoginForm` 内部管理，而不是提升到 `App`。

# 第十三章：WebSocket 协议详解

## 13.1 WebSocket 基础

### 13.1.1 什么是 WebSocket

WebSocket 是一种在单个 TCP 连接上进行全双工通信的协议。与 HTTP 的请求-响应模式不同，WebSocket 允许服务器主动向客户端推送数据。

**HTTP vs WebSocket：**

```
HTTP:
Client → Request → Server
Client ← Response ← Server

WebSocket:
Client ←→ Data ←→ Server (双向、持续)
```

### 13.1.2 WebSocket 握手

WebSocket 连接通过 HTTP 升级请求建立：

```
GET /ws/terminal HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

握手完成后，连接升级为 WebSocket 协议，双方可以自由发送数据。

### 13.1.3 WebSocket 消息类型

WebSocket 支持两种消息类型：

- **文本消息** — UTF-8 编码的字符串
- **二进制消息** — ArrayBuffer 或 Blob

本项目中：
- **终端通道** — 使用二进制消息（终端输出数据）
- **控制通道** — 使用文本消息（JSON 格式的控制消息）

## 13.2 本项目的 WebSocket 协议

### 13.2.1 协议版本

```typescript
import { PROTOCOL_VERSION } from '@ai-cli/shared'
```

协议版本用于客户端和服务器的版本协商。如果版本不匹配，服务器会关闭连接并返回 `WS_CLOSE_CODE.PROTOCOL_MISMATCH` 关闭码。

### 13.2.2 认证流程

```
Client                          Server
  |                               |
  |--- AUTH (accessToken) -------->|
  |<-- AUTH_OK --------------------|
  |                               |
  |--- ATTACH_SESSION (sessionId)->|  (终端通道)
  |--- INIT_SESSION (sessionId)-->|  (控制通道)
  |<-- SESSION_READY --------------|
  |                               |
  |<=== 二进制数据 / JSON 消息 ===>|
```

### 13.2.3 终端通道消息

**客户端 → 服务器：**
- 二进制数据 — 用户的键盘输入
- 单字节 `TERM_PING` — 心跳包

**服务器 → 客户端：**
- 二进制数据 — 终端输出（ANSI 转义序列）
- 单字节 `0x01` — 心跳响应

### 13.2.4 控制通道消息

**客户端 → 服务器：**

```typescript
type ControlClientMessage =
  | { type: 'AUTH'; accessToken: string; protocolVersion: number }
  | { type: 'INIT_SESSION'; sessionId: string; cols: number; rows: number; adapter: string }
  | { type: 'ATTACH_SESSION'; sessionId: string }
  | { type: 'RESIZE'; sessionId: string; cols: number; rows: number }
  | { type: 'QUICK_ACTION'; sessionId: string; payload: string }
  | { type: 'INJECT_CODE'; sessionId: string; code: string }
  | { type: 'PING' }
  | { type: 'OBSERVE_SESSION'; sessionId: string }
```

**服务器 → 客户端：**

```typescript
type ControlServerMessage =
  | { type: 'AUTH_OK' }
  | { type: 'SESSION_READY' }
  | { type: 'STATUS_UPDATE'; sessionId: string; status: AgentStatus }
  | { type: 'TOKEN_RENEWED'; accessToken: string }
  | { type: 'ERROR'; message: string }
  | { type: 'PONG' }
```

### 13.2.5 关闭码

```typescript
const WS_CLOSE_CODE = {
  PROTOCOL_MISMATCH: 4000,  // 协议版本不匹配
  AUTH_FAILED: 4001,         // 认证失败
}
```

当收到这些关闭码时，客户端需要采取不同的措施：
- `4000` — 刷新页面，加载新版本
- `4001` — 尝试刷新 token，失败则登出

## 13.3 心跳机制

### 13.3.1 为什么需要心跳

WebSocket 连接可能因为以下原因断开：

1. **网络中断** — 用户的网络连接不稳定
2. **NAT 超时** — 路由器或防火墙可能会关闭空闲连接
3. **服务器超时** — 服务器可能会关闭空闲连接
4. **浏览器限制** — 浏览器可能会限制后台标签页的连接

心跳机制通过定期发送小数据包来保持连接活跃。

### 13.3.2 本项目的心跳实现

```tsx
// 终端心跳 — 每 30 秒发送一个字节
function startTermPing() {
  termPingRef.current = setInterval(() => {
    const ws = termWsRef.current
    if (ws && ws.readyState === WebSocket.OPEN) {
      ws.send(new Uint8Array([TERM_PING]))
    }
  }, 30_000)
}

// 控制心跳 — 每 30 秒发送 JSON PING
function startCtrlPing() {
  ctrlPingRef.current = setInterval(() => {
    const ws = ctrlWsRef.current
    if (ws && ws.readyState === WebSocket.OPEN) {
      ws.send(JSON.stringify({ type: 'PING' }))
    }
  }, 30_000)
}
```

**心跳检测：**

终端通道收到单字节 `0x01` 时忽略（不写入终端）。控制通道收到 `PONG` 消息时忽略。

## 13.4 重连策略

### 13.4.1 指数退避（Exponential Backoff）

指数退避是一种重试策略，每次重试的等待时间呈指数增长：

```
第 1 次重连：1 秒后
第 2 次重连：2 秒后
第 3 次重连：4 秒后
第 4 次重连：8 秒后
第 5 次重连：16 秒后
第 6 次重连：30 秒后（上限）
```

### 13.4.2 随机抖动（Jitter）

随机抖动在指数退避的基础上添加随机性：

```tsx
const jittered = delay * (0.5 + Math.random() * 0.5)
```

这意味着实际的等待时间在 `delay * 0.5` 到 `delay * 1.0` 之间随机分布。

**为什么需要抖动？**

假设 1000 个客户端同时断线，没有抖动的话它们会在同一时刻重连（1s、2s、4s...），造成服务器瞬间高峰。抖动使重连时间分散开来，平滑服务器负载。

### 13.4.3 独立通道重连

本项目支持终端通道和控制通道的独立重连：

```tsx
function reconnectTermOnly() {
  // 只关闭终端通道
  // 保持控制通道
  // 重新连接终端通道
}

function reconnectCtrlOnly() {
  // 只关闭控制通道
  // 保持终端通道
  // 重新连接控制通道
}
```

这种设计提高了连接的可靠性——即使一个通道断开，另一个通道仍然可以正常工作。

# 第十七章：代码逐行深度解析 — 所有文件完整版

## 17.1 main.tsx 完整逐行解析

让我们从最基础的入口文件开始，逐字符地分析每一行代码的含义。

### 第 1 行：`import React from 'react'`

这行代码从 `react` 包中导入默认导出 `React`。`react` 包是 React 库的核心，包含了创建元素、组件、Hook 等所有核心 API。

在模块系统中，`import` 语句会被打包工具（Vite/Webpack）解析。打包工具会：
1. 在 `node_modules` 中找到 `react` 包
2. 读取 `package.json` 中的 `main` 或 `module` 字段
3. 加载对应的入口文件
4. 将导入的绑定与导出的值关联起来

在 React 17+ 的新 JSX 转换中，不再需要显式导入 `React`。编译器会自动从 `react/jsx-runtime` 导入必要的函数。但在使用 `React.StrictMode` 时仍然需要。

### 第 2 行：`import ReactDOM from 'react-dom/client'`

从 `react-dom/client` 子路径导入 ReactDOM。React 18 将 `createRoot` API 移到了 `react-dom/client` 子路径下，以区分新旧 API。

`react-dom` 是 React 的 DOM 渲染器，负责将 React 元素转换为真实的 DOM 节点。它还处理事件委托、DOM 更新等。

### 第 3 行：`import App from './App'`

从当前目录的 `App` 文件导入默认导出。Vite 会自动解析 `.tsx`、`.ts`、`.jsx`、`.js` 等扩展名。

这里使用的是相对路径（以 `./` 开头），表示同目录下的文件。模块解析顺序：
1. `./App.tsx` — TypeScript JSX 文件
2. `./App.ts` — TypeScript 文件
3. `./App.jsx` — JavaScript JSX 文件
4. `./App.js` — JavaScript 文件
5. `./App/index.tsx` — 目录下的 index 文件

### 第 4 行：`import { requestNotificationPermission } from './lib/notifications'`

从 `./lib/notifications` 模块导入具名导出 `requestNotificationPermission`。这是项目自定义的通知权限请求函数。

注意这里使用的是具名导入（`{ ... }`），而不是默认导入。一个模块可以有多个具名导出，但只能有一个默认导出。

### 第 5 行：`import './index.css'`

导入 CSS 文件。这是一个副作用导入（side-effect import），没有导入任何绑定，只是让打包工具处理这个文件。

Vite 处理 CSS 导入的方式：
1. 在开发模式下，CSS 通过 `<style>` 标签注入到页面中，支持热模块替换（HMR）
2. 在生产模式下，CSS 被提取到独立的 `.css` 文件中

### 第 7 行：`requestNotificationPermission()`

这是一个模块级别的函数调用。当这个模块被加载时，这个调用就会执行。

**执行时机：** 当 `main.tsx` 被 Vite 的开发服务器加载时，或者当打包后的 `main-[hash].js` 被浏览器加载时。

**为什么不在 useEffect 中调用？**

1. 这个操作只需要执行一次
2. 不需要与组件生命周期绑定
3. 在模块加载时执行是最简单、最直接的方式
4. 如果放在 `useEffect` 中，需要等到组件挂载后才执行，可能错过最佳时机

### 第 9-12 行：渲染入口

```tsx
ReactDOM.createRoot(document.getElementById('root')!).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>,
)
```

**`document.getElementById('root')`** — 从 DOM 中查找 id 为 `root` 的元素。这个元素在 `index.html` 中定义。

**`!`（非空断言）** — 告诉 TypeScript 编译器这个值不会是 `null`。如果运行时确实为 `null`，会抛出运行时错误。

**`ReactDOM.createRoot()`** — 创建一个并发模式的 React 根节点。返回一个 `Root` 对象。

**`.render()`** — 将 React 元素渲染到根节点中。这个调用是异步的——React 会在下一个微任务中开始渲染。

**`<React.StrictMode>`** — 严格模式组件，在开发模式下：
- 组件函数会被调用两次（检测副作用）
- `useEffect` 的清理函数会被调用两次
- `useState` 的 `setState` 函数会被调用两次

这些额外的调用帮助开发者发现不纯的组件函数和未清理的副作用。

### 懒加载 CodeEditor

```tsx
const CodeEditor = lazy(() => import('./components/CodeEditor').then(m => ({ default: m.CodeEditor })))
```

**逐层解析：**

1. `import('./components/CodeEditor')` — 动态导入。Vite 会将这个模块分割成独立的 chunk。返回一个 `Promise`。

2. `.then(m => ({ default: m.CodeEditor }))` — 当 Promise 解析后，`m` 是整个模块对象，包含所有具名导出。`m.CodeEditor` 是具名导出的 CodeEditor 组件。我们需要返回 `{ default: m.CodeEditor }`，因为 `React.lazy` 要求模块有 `default` 导出。

3. `lazy(() => ...)` — `React.lazy` 接受一个工厂函数，该函数返回一个 `Promise`。当组件首次被渲染时，React 会调用这个工厂函数来加载组件。

**加载流程：**

1. 用户打开应用 → CodeEditor 不会被加载
2. 用户在文件浏览器中选择一个文件 → `setEditorFile(...)` 被调用
3. React 尝试渲染 `<CodeEditor>` → `lazy` 的工厂函数被调用
4. 浏览器下载 CodeMirror chunk → 加载完成
5. React 渲染 CodeEditor 组件

**Fallback UI：**

```tsx
<Suspense fallback={
  <div className="fixed inset-0 z-30 bg-[#1a1b26] flex items-center justify-center text-gray-400 text-sm">
    Loading editor...
  </div>
}>
  <CodeEditor ... />
</Suspense>
```

`<Suspense>` 组件在 CodeEditor 加载完成前显示 fallback UI。这是一个全屏的加载提示，使用与编辑器相同的背景色，避免闪烁。

### 状态声明

```tsx
const [username, setUsername] = useState('')
const [password, setPassword] = useState('')
const [error, setError] = useState('')
const [loading, setLoading] = useState(false)
```

**useState 的内部实现原理（简化版）：**

```typescript
let state = []
let cursor = 0

function useState(initialValue) {
  const currentCursor = cursor
  if (state[currentCursor] === undefined) {
    state[currentCursor] = initialValue
  }
  
  function setState(newValue) {
    state[currentCursor] = newValue
    // 触发重新渲染
    scheduleRerender()
  }
  
  cursor++
  return [state[currentCursor], setState]
}
```

React 通过调用顺序来维护状态。第一次 `useState` 调用对应 `state[0]`，第二次对应 `state[1]`，以此类推。这就是为什么 Hook 不能在条件语句中调用——条件会改变调用顺序。

### 表单提交处理

```tsx
async function handleSubmit(e: React.FormEvent) {
  e.preventDefault()
  setError('')
  setLoading(true)
  try {
    await onLogin(username, password)
  } catch (err) {
    setError(err instanceof Error ? err.message : '登录失败')
  } finally {
    setLoading(false)
  }
}
```

**`React.FormEvent`** — React 的表单事件类型。它是 `SyntheticEvent` 的子类型，提供了表单特定的属性。

**`e.preventDefault()`** — 阻止表单的默认提交行为。在 HTML 中，表单提交会导致页面跳转。在 SPA 中，我们不希望页面跳转，所以需要阻止默认行为。

**`try/catch/finally` 模式：**

```
try {
  // 尝试执行可能失败的操作
  await onLogin(username, password)
} catch (err) {
  // 如果失败，处理错误
  setError(...)
} finally {
  // 无论成功还是失败，都会执行
  setLoading(false)
}
```

`finally` 块确保 `loading` 状态在操作完成后被清除，无论操作是成功还是失败。

### onInjectCode 回调


**逐层解析：**

1. `useSessionStore.getState()` — 获取 Zustand store 的当前状态

### closeSockets 详解

```tsx
function closeSockets() {
  if (termWsRef.current) {
    termWsRef.current.onopen = null
    termWsRef.current.onmessage = null
    termWsRef.current.onclose = null
    termWsRef.current.onerror = null
    if (termWsRef.current.readyState === WebSocket.OPEN || termWsRef.current.readyState === WebSocket.CONNECTING) {
      termWsRef.current.close()
    }
    termWsRef.current = null
  }
  if (ctrlWsRef.current) {
    ctrlWsRef.current.onopen = null
    ctrlWsRef.current.onmessage = null
    ctrlWsRef.current.onclose = null
    ctrlWsRef.current.onerror = null
    if (ctrlWsRef.current.readyState === WebSocket.OPEN || ctrlWsRef.current.readyState === WebSocket.CONNECTING) {
      ctrlWsRef.current.close()
    }
    ctrlWsRef.current = null
  }
}
```

**WebSocket readyState 状态：**

- `WebSocket.CONNECTING (0)` — 连接正在建立
- `WebSocket.OPEN (1)` — 连接已建立，可以通信
- `WebSocket.CLOSING (2)` — 连接正在关闭
- `WebSocket.CLOSED (3)` — 连接已关闭

只有在 `OPEN` 或 `CONNECTING` 状态时才需要调用 `close()`。如果已经在 `CLOSING` 或 `CLOSED` 状态，调用 `close()` 是无害的，但没有必要。

### scheduleReconnect 详解

```tsx
function scheduleReconnect() {
  if (reconnectTimerRef.current) return

  const delay = reconnectDelayRef.current
  const jittered = delay * (0.5 + Math.random() * 0.5)
  reconnectDelayRef.current = Math.min(delay * 2, MAX_RECONNECT_DELAY)

  reconnectTimerRef.current = setTimeout(() => {
    reconnectTimerRef.current = null
    const s = sessionRef.current
    const t = termRef.current
    if (s && t) {
      setReconnectCount((c) => c + 1)
      connectInternal(s.sessionId, s.cols, s.rows, t)
    }
  }, jittered)
}
```

**指数退避计算：**

```
初始: delay = 1000
第 1 次: jittered = 1000 * (0.5 + random * 0.5) = 500 ~ 1000ms
         delay = min(1000 * 2, 30000) = 2000
第 2 次: jittered = 2000 * (0.5 + random * 0.5) = 1000 ~ 2000ms
         delay = min(2000 * 2, 30000) = 4000
第 3 次: jittered = 4000 * (0.5 + random * 0.5) = 2000 ~ 4000ms
         delay = min(4000 * 2, 30000) = 8000
...
第 5 次: delay = min(16000 * 2, 30000) = 30000
第 6 次: delay = 30000 (不再增长)
```

**定时器清理：**

`reconnectTimerRef.current = null` 在定时器回调中清除引用，确保下一次 `scheduleReconnect` 调用可以正常工作。

### sendResize 详解

```tsx
const sendResize = useCallback((cols: number, rows: number) => {
  if (resizeDebounceRef.current) {
    clearTimeout(resizeDebounceRef.current)
  }

  resizeDebounceRef.current = setTimeout(() => {
    const now = Date.now()
    if (now - lastResizeSentRef.current < RESIZE_THROTTLE) return
    lastResizeSentRef.current = now

    const ws = ctrlWsRef.current
    const sessionId = sessionRef.current?.sessionId
    if (ws && ws.readyState === WebSocket.OPEN && sessionId) {
      const msg: ControlClientMessage = { type: 'RESIZE', sessionId, cols, rows }
      ws.send(JSON.stringify(msg))
    }

    if (sessionRef.current) {
      sessionRef.current.cols = cols
      sessionRef.current.rows = rows
    }
  }, RESIZE_DEBOUNCE)
}, [])
```

**防抖 + 节流的组合：**

```
用户快速调整窗口大小 → 多次触发 resize 事件
                    ↓
防抖：等待 200ms 没有新事件后才执行
                    ↓
节流：即使防抖通过，也限制至少 1 秒才能发送一次
                    ↓
发送 resize 消息到服务器
```

**为什么需要双重优化？**

- 只用防抖：如果用户持续调整窗口大小（超过 200ms 的间隔），会频繁发送消息
- 只用节流：在节流窗口内可能会发送多余的消息
- 组合使用：防抖确保只在用户停止调整后才发送，节流确保发送频率有上限

# 第十八章：Monorepo 架构

## 18.1 什么是 Monorepo

Monorepo（单一仓库）是一种代码管理策略，将所有项目代码放在一个版本控制仓库中。

**Monorepo vs Polyrepo：**

```
Monorepo:
repo/
├── apps/
│   ├── web/          # 前端
│   └── server/       # 后端
├── packages/
│   ├── shared/       # 共享代码
│   └── config/       # 共享配置
└── package.json

Polyrepo:
web-repo/             # 前端仓库
server-repo/          # 后端仓库
shared-repo/          # 共享代码仓库
```

### 18.1.1 Monorepo 的优势

1. **代码共享** — 多个项目可以轻松共享代码
2. **原子提交** — 一次提交可以同时修改多个项目
3. **统一版本** — 所有项目使用相同的依赖版本
4. **简化 CI/CD** — 只需要一个构建管道
5. **更好的代码审查** — 所有代码在一个地方

### 18.1.2 Monorepo 的挑战

1. **仓库体积** — 随着项目增长，仓库会变得很大
2. **构建时间** — 需要增量构建来优化
3. **权限管理** — 难以限制对特定目录的访问
4. **工具复杂性** — 需要专门的工具（如 Turborepo、Nx、pnpm workspaces）

## 18.2 本项目的 Monorepo 结构


# 第十九章：构建工具 — Vite 深入

## 19.1 Vite 的工作原理

### 19.1.1 开发模式

Vite 在开发模式下使用**原生 ES 模块**（Native ESM）来提供开发服务器。它不需要像 Webpack 那样将所有模块打包成一个文件。

**工作流程：**

1. 浏览器请求 `index.html`
2. Vite 返回 `index.html`，其中包含 `<script type="module" src="/src/main.tsx">`
3. 浏览器请求 `/src/main.tsx`
4. Vite 编译 TypeScript/JSX，返回 JavaScript
5. 浏览器解析 `import` 语句，请求依赖模块
6. Vite 按需编译和返回每个模块

**优势：**

- **快速启动** — 不需要打包整个应用
- **快速热更新** — 只更新变化的模块
- **低内存使用** — 不需要维护完整的打包图

### 19.1.2 生产模式

在生产模式下，Vite 使用 Rollup 来打包代码。Rollup 是一个 JavaScript 模块打包器，擅长生成高效的代码。

**打包流程：**

1. **解析** — 从入口文件开始，解析所有 `import` 语句
2. **转换** — 使用插件转换代码（TypeScript、JSX、CSS 等）
3. **Tree Shaking** — 移除未使用的代码
4. **代码分割** — 将代码分割成多个 chunk
5. **优化** — 压缩、混淆代码
6. **输出** — 生成最终的文件

### 19.1.3 插件系统

Vite 的插件系统兼容 Rollup 的插件接口，同时添加了一些 Vite 特有的钩子。

**常用插件：**

- `@vitejs/plugin-react` — React 支持（JSX 转换、快速刷新）

# 第二十一章：深入理解每个函数

## 21.1 formatFileSize 函数深度解析

```typescript
function formatFileSize(bytes: number): string {
  if (bytes < 1024) return `${bytes} B`
  if (bytes < 1024 * 1024) return `${(bytes / 1024).toFixed(1)} KB`
  return `${(bytes / (1024 * 1024)).toFixed(1)} MB`
}
```

### 21.1.1 函数签名分析

**`formatFileSize`** — 函数名，使用驼峰命名法，表示"格式化文件大小"。

**`(bytes: number)`** — 参数列表。`bytes` 是文件大小的字节数，类型为 `number`。TypeScript 的类型注解确保调用者传入正确的类型。

**`: string`** — 返回类型。函数返回一个格式化后的字符串。

### 21.1.2 函数体逐行分析

**第 1 行：`if (bytes < 1024) return `${bytes} B``**

检查文件大小是否小于 1024 字节（1 KB）。如果是，直接返回字节数加上 ` B` 后缀。

**模板字符串：** `` `${bytes} B` `` 使用 ES6 的模板字符串语法。`${bytes}` 会被替换为 `bytes` 变量的值。

**第 2 行：`if (bytes < 1024 * 1024) return `${(bytes / 1024).toFixed(1)} KB``**

检查文件大小是否小于 1048576 字节（1 MB）。如果是，将字节数除以 1024 得到 KB 数。

**`.toFixed(1)`** — 将数字格式化为保留 1 位小数的字符串。例如：`(1024 / 1024).toFixed(1)` = `"1.0"`。

**第 3 行：`return `${(bytes / (1024 * 1024)).toFixed(1)} MB``**

对于大于等于 1 MB 的文件，将字节数除以 1048576 得到 MB 数。

### 21.1.3 为什么使用 1024 而不是 1000

在计算机科学中，数据存储单位通常使用 2 的幂次：
- 1 KB = 1024 字节（2^10）
- 1 MB = 1048576 字节（2^20）
- 1 GB = 1073741824 字节（2^30）

这与国际单位制（SI）不同，SI 使用 10 的幂次：
- 1 kB = 1000 字节
- 1 MB = 1000000 字节

在文件系统和网络传输中，通常使用 1024 进制。

### 21.1.4 函数的局限性

这个函数没有处理 GB 及以上的单位。如果文件大小超过 1 GB，会显示为一个很大的 MB 数。更完善的实现应该支持 GB、TB 等单位：

```typescript
function formatFileSize(bytes: number): string {
  const units = ['B', 'KB', 'MB', 'GB', 'TB']
  let unitIndex = 0
  let size = bytes

  while (size >= 1024 && unitIndex < units.length - 1) {
    size /= 1024
    unitIndex++
  }

  return `${size.toFixed(1)} ${units[unitIndex]}`
}
```

## 21.2 buildBreadcrumbs 函数深度解析

```typescript
function buildBreadcrumbs(path: string): { label: string; path: string }[] {
  if (!path) return [{ label: '~', path: '' }]
  const parts = path.split('/').filter(Boolean)
  const crumbs = [{ label: '~', path: '' }]
```

### 21.2.1 函数签名

### 21.2.3 路径分割

**`path.split('/')`** — 使用 `/` 分割路径字符串。例如：`'home/user/docs'` → `['home', 'user', 'docs']`。

**`.filter(Boolean)`** — 过滤掉空字符串。这处理了路径以 `/` 开头或结尾的情况。例如：`'/home/user/'` → `['', 'home', 'user', '']` → `['home', 'user']`。

### 21.2.4 面包屑构建

```typescript
const crumbs = [{ label: '~', path: '' }]
let accumulated = ''

  crumbs.push({ label: part, path: accumulated })
```

**累加路径：**

对于路径 `'home/user/docs'`：

| 迭代 | part | accumulated | crumbs |
|------|------|-------------|--------|
| 初始 | - | '' | [{label:'~', path:''}] |

| 2 | 'user' | 'home/user' | [..., {label:'user', path:'home/user'}] |
| 3 | 'docs' | 'home/user/docs' | [..., {label:'docs', path:'home/user/docs'}] |

## 21.4 parseJwtExp 函数深度解析

```typescript
function parseJwtExp(token: string): number {
  try {
    const payload = token.split('.')[1]
    const decoded = JSON.parse(atob(payload))
    return decoded.exp * 1000
  } catch {
    return 0
  }
}
```

### 21.4.1 JWT 结构

JWT（JSON Web Token）由三部分组成，用 `.` 分隔：

```
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

- **Payload** — 数据（用户信息、过期时间等）
- **Signature** — 签名

### 21.4.2 逐行分析

**`token.split('.')[1]`** — 提取 Payload 部分。`split('.')` 使用 `.` 分割字符串，取索引为 1 的元素。

**`atob(payload)`** — Base64 解码。JWT 的 Payload 是 Base64 编码的 JSON 字符串。

**`JSON.parse(decoded)`** — 将 JSON 字符串解析为 JavaScript 对象。

**`decoded.exp * 1000`** — `exp` 是 JWT 的过期时间字段，单位是 Unix 秒。乘以 1000 转换为 JavaScript 的毫秒时间戳。

```typescript
private handleCompositionEnd(e: CompositionEvent): void {
```

## 21.6 getDistance 函数深度解析

```typescript
private getDistance(t1: Touch, t2: Touch): number {
  const dx = t1.clientX - t2.clientX
  const dy = t1.clientY - t2.clientY
  return Math.sqrt(dx * dx + dy * dy)
}
```

### 21.6.1 欧几里得距离

这个函数计算两个触摸点之间的欧几里得距离。公式：

```
distance = √((x₂ - x₁)² + (y₂ - y₁)²)
```

### 21.6.2 逐行分析

**`const dx = t1.clientX - t2.clientX`** — 计算 X 轴上的距离。

**`const dy = t1.clientY - t2.clientY`** — 计算 Y 轴上的距离。

**`Math.sqrt(dx * dx + dy * dy)`** — 计算欧几里得距离。`dx * dx` 是 `dx` 的平方，`Math.sqrt` 是平方根。

### 21.6.3 Touch 接口

`Touch` 接口表示一个触摸点：

- `clientX` — 触摸点相对于视口的 X 坐标
- `clientY` — 触摸点相对于视口的 Y 坐标
- `pageX` — 触摸点相对于页面的 X 坐标
- `pageY` — 触摸点相对于页面的 Y 坐标
- `screenX` — 触摸点相对于屏幕的 X 坐标
- `screenY` — 触摸点相对于屏幕的 Y 坐标
- `identifier` — 触摸点的唯一标识

# 第二十二章：错误处理策略

## 22.1 前端错误分类

### 22.1.1 语法错误（Syntax Error）

语法错误在编译时被捕获，TypeScript 编译器会指出这些错误。

```typescript
// ❌ 语法错误
const x = { name: "test" }
console.log(x.namee)  // TypeScript: 属性 'namee' 不存在

// ❌ 类型错误
const y: number = "hello"  // TypeScript: 不能将 string 赋值给 number
```

### 22.1.2 运行时错误（Runtime Error）

运行时错误在代码执行时发生，需要通过 `try/catch` 或 ErrorBoundary 捕获。

```typescript
// 空指针错误
const obj = null
console.log(obj.name)  // TypeError: Cannot read property 'name' of null

// 网络错误
const res = await fetch('/api/data')  // 可能抛出网络错误
```

### 22.1.3 逻辑错误（Logic Error）

逻辑错误不会抛出异常，但会导致不正确的结果。

```typescript
// 逻辑错误：应该用 === 而不是 ==
if (value == null) { ... }  // 可能不是预期的行为
```

## 22.2 本项目的错误处理模式

### 22.2.1 异步操作的错误处理

```typescript
async function handleSubmit(e: React.FormEvent) {
  e.preventDefault()
  setError('')
  setLoading(true)
  try {
    await onLogin(username, password)
  } catch (err) {
    setError(err instanceof Error ? err.message : '登录失败')
  } finally {
    setLoading(false)
  }
}
```

**模式：try/catch/finally**

- `try` — 包裹可能失败的异步操作
- `catch` — 捕获错误，显示用户友好的错误信息
- `finally` — 无论成功还是失败，都清除加载状态

### 22.2.2 API 请求的错误处理

```typescript
const res = await fetch(url, options)
if (!res.ok) {
  const data = await res.json().catch(() => ({ error: 'Request failed' }))
  throw new Error(data.error || `HTTP ${res.status}`)
}
```

**模式：检查响应状态**

- `res.ok` — HTTP 状态码是否在 200-299 范围内
- `.catch(() => ...)` — 如果解析 JSON 失败，使用默认错误信息
- `throw new Error(...)` — 抛出错误，由调用者处理

### 22.2.3 WebSocket 错误处理

```typescript
termWs.onclose = (event) => {
  if (event.code === WS_CLOSE_CODE.PROTOCOL_MISMATCH) {
    window.location.reload()
    return
  }
  if (event.code === WS_CLOSE_CODE.AUTH_FAILED) {
    handleAuthFailureAndRetry()
    return
  }
  scheduleReconnect()
}
```

**模式：根据关闭码处理**

- `PROTOCOL_MISMATCH` — 刷新页面，加载新版本
- `AUTH_FAILED` — 尝试刷新 token
- 其他 — 调度重连

## 22.3 错误边界（Error Boundary）

React 的错误边界是一种组件，可以捕获其子组件中的 JavaScript 错误，记录错误，并显示备用 UI。

```tsx
class ErrorBoundary extends React.Component {
  constructor(props) {
    super(props)
    this.state = { hasError: false, error: null }
  }

  static getDerivedStateFromError(error) {
    return { hasError: true, error }
  }

  componentDidCatch(error, errorInfo) {
    console.error('Error caught by boundary:', error, errorInfo)
  }

  render() {
    if (this.state.hasError) {
      return <div>Something went wrong: {this.state.error?.message}</div>
    }
    return this.props.children
  }
}
```

**本项目未使用错误边界**，因为组件结构相对简单，且主要的错误都通过 try/catch 处理。

# 第二十四章：安全考虑

## 24.1 认证安全

### 24.1.1 Token 存储

本项目使用 `localStorage` 存储 JWT token：

```typescript
function storeTokens(access: string, refresh: string) {
  localStorage.setItem(TOKEN_KEY, JSON.stringify({ accessToken: access, refreshToken: refresh }))
}
```

**安全考虑：**

- `localStorage` 是持久的，即使关闭浏览器也不会清除
- `localStorage` 可以被同源的任何 JavaScript 代码访问
- 如果应用存在 XSS 漏洞，攻击者可以窃取 token

**更安全的替代方案：**

- **httpOnly Cookie** — 不能被 JavaScript 访问
- **内存存储** — 关闭浏览器后自动清除（但不持久）
- **Service Worker 存储** — 更安全，但实现复杂

### 24.1.2 Token 刷新

```typescript
const scheduleTokenRenewal = useCallback((token: string) => {
  const expiresAt = parseJwtExp(token)
  const refreshIn = expiresAt - Date.now() - 2 * 60 * 1000
  if (refreshIn <= 0) return
  refreshTimerRef.current = setTimeout(async () => {
    try {
      await doRefreshTokenRef.current?.()
    } catch { /* 静默失败 */ }
  }, refreshIn)
}, [])
```

**安全考虑：**

- 在 token 过期前 2 分钟刷新，确保 token 在使用过程中不会过期
- 如果刷新失败，静默处理——WebSocket 的 4001 关闭码会触发重新认证

### 24.1.3 XSS 防护

React 默认对 JSX 中的内容进行转义，防止 XSS 攻击：

```tsx
// React 自动转义
<p>{userInput}</p>  // <script>alert('xss')</script> 被转义为文本
```

**危险的用法：**

```tsx
// ❌ 危险 — 直接设置 innerHTML
<div dangerouslySetInnerHTML={{ __html: userInput }} />

// ✅ 安全 — React 自动转义
<div>{userInput}</div>
```
